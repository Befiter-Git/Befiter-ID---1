🏗️ Recommended architecture: Webhook + Reconciliation Poll (hybrid)
This is the same pattern used by Stripe, Shopify, Linear, GitHub — proven at scale. Single mechanisms (pure webhook OR pure polling) each have failure modes; combining them is what makes it bulletproof.

                  ┌─────────────────┐
                  │  befiter.store  │  (event source of truth)
                  └────────┬────────┘
                           │
        ┌──────────────────┴──────────────────┐
        ▼                                     ▼
   1. WEBHOOK (real-time, < 1 sec)      2. RECONCILIATION POLL (safety net, every 30 min)
   POST /api/webhooks/befiter-store     GET /api/s2s/member?befiterId=… (we already have this)
        │                                     │
        └──────────────────┬──────────────────┘
                           ▼
                  ┌─────────────────┐
                  │  befiter.com    │
                  │  (event log +   │
                  │   local cache)  │
                  └─────────────────┘

Why hybrid (not just webhook)
A webhook can be missed (network blip, our server restarting, store retry exhausted).
The 30-min reconciliation poll guarantees that within ~30 min the local copy matches store, even if every webhook for that member failed.
For active members the user-perceived freshness is still real-time (webhooks deliver in seconds); the poll is invisible insurance.

Standard envelope (every event has this shape)
{
  "eventId": "evt_01HX8K…",
  "eventType": "membership.created",
  "occurredAt": "2026-05-03T10:00:00Z",
  "version": 1,
  "befiterId": "abc-123",
  "studioId": "studio-andheri",
  "data": { …event-specific… }

Required headers
Header	Purpose
X-Befiter-Store-Event-Id	Idempotency key (matches eventId)
X-Befiter-Store-Signature: sha256=…	HMAC of raw body using shared secret
X-Befiter-Store-Timestamp	Unix epoch — we reject events > 5 min old (replay protection)
🛠️ What we build on the befiter.com side (3 layers)
Layer 1: The webhook receiver (POST /api/webhooks/befiter-store)
HMAC signature verification (constant-time compare)
Timestamp window check (reject stale)
eventId deduplication via a new webhook_events table with unique index
Push-then-ack pattern: write event to durable log first, return 200 immediately, process async — so a slow handler can't cause the store to retry-storm us
Rate limiter
Layer 2: Event log + per-event handlers
New table webhook_events — every event ever received, full payload, processing status (received / processed / failed / dead), retry count, error
One handler function per eventType, all in server/api/webhooks/befiter-store-handlers.ts
Each handler is idempotent — running it twice on the same event yields the same result
Layer 3: Local domain tables (the data the UI reads)
We'd extend the local schema to mirror the store data we care about. Two options:

Approach	Pros	Cons
A. Side tables per concept (member_memberships, member_freezes, member_bookings, member_attendance, member_checkins)	Queryable, indexable, joinable. Powers dashboard widgets, history pages, analytics.	More schema; needs migrations.
B. Single member_events table with JSON payload	Minimal schema. Append-only audit log.	Hard to query ("show my last 5 bookings" needs JSON path queries).
My strong recommendation: A for memberships/freezes/bookings (frequently queried), B for attendance/check-ins as audit log + a denormalized "last_checkin_at" on member_profiles for the card.

Layer 4: Reconciliation poll
Cron every 30 min walks every member with an active befiterId and calls /api/s2s/member?befiterId=X
Compares to local cache; if anything differs, treats it as a synthetic event and runs the same handler logic (so any drift gets repaired, missed events recovered)
Backs off members with no recent activity to save calls
